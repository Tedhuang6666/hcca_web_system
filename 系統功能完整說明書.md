## 一、系統總覽

### 1.1 系統定位

**校園自治整合平台**是服務學生代表大會（HCCA）的數位治理工具，將校園自治所需的行政作業全面數位化。系統設計哲學以**五院分立**（立法、行政、司法、考試、監察）為模組職責邊界。

### 1.2 技術架構

| 層次 | 技術 | 說明 |
|------|------|------|
| 後端 API | FastAPI + Python 3.12 | 非同步優先（Async-first） |
| 資料庫 | PostgreSQL 16 | 主要資料儲存 |
| 快取/佇列 | Redis 7 | Token 黑名單、Celery 佇列 |
| 背景任務 | Celery 5.3 | Email 發送、定時結單 |
| 前端 | Next.js 16 + React 19 | App Router 架構 |
| 樣式 | Tailwind CSS 4 | Utility-first |
| 認證 | Google OAuth2 + JWT | 雙 Token 機制 |
| 即時通訊 | WebSocket | 房間訂閱制 |

### 1.3 核心功能模組一覽

| 模組 | 狀態 | 主要功能 |
|------|------|---------|
| 身份驗證 | ✅ | Google OAuth2、JWT 雙 Token、Token 黑名單 |
| 組織架構 | ✅ | 組織樹、職位、時間有效任期 |

## 二、身份驗證與帳號管理

### 2.1 登入流程

系統採用 **Google OAuth2** 作為唯一登入方式，不支援帳號密碼登入。

**完整登入步驟：**

1. 使用者點擊「使用 Google 登入」按鈕
2. 瀏覽器重導向至 Google 授權頁（`GET /auth/google/login`）
3. 使用者在 Google 頁面同意授權
4. Google 回呼後端（`GET /auth/google/callback`）
5. 系統自動：
   - 若帳號不存在：自動建立新帳號
   - 若帳號已存在：更新 Google 資料（名稱、頭像）
   - 從學校信箱格式 `g0{學號}@hchs.hc.edu.tw` 自動解析學號
   - 若信箱在 `SUPERUSER_EMAILS` 環境變數中，自動設為超級管理員
6. 產生 Access Token 和 Refresh Token，重導向前端 `/auth/callback`
7. 前端儲存 Token 至 localStorage

### 2.2 Token 機制

| Token 類型 | 用途 | 有效期 | 儲存位置 |
|-----------|------|--------|---------|
| **Access Token** | API 請求認證（Bearer Token） | 短效（分鐘級） | localStorage |
| **Refresh Token** | 換發新 Access Token | 長效（天/週級） | localStorage |

**Token 刷新（`POST /auth/refresh`）：**
- 輸入：舊 Refresh Token
- 輸出：新的 Access Token + 新的 Refresh Token
- 舊 Refresh Token 自動加入 Redis 黑名單（Token Rotation 機制）

**登出（`POST /auth/logout`）：**
- 當前 Access Token 加入 Redis 黑名單
- 客戶端清除 localStorage 中的 Token

### 2.3 使用者資料

**查看自己的資訊（`GET /auth/me`）：**

回傳欄位包含：
- `id`：使用者 UUID
- `email`：Google 信箱
- `display_name`：顯示名稱
- `avatar_url`：Google 頭像網址
- `student_id`：學號（可手動填入或自動從信箱解析）
- `phone`：電話（選填）
- `show_phone`：是否公開電話
- `show_email`：是否公開信箱
- `is_superuser`：是否為超級管理員
- `permissions`：當前有效的所有權限碼列表

### 2.4 個人資料管理

**搜尋使用者（`GET /users`）：**
- 支援以名稱或學號關鍵字搜尋
- 主要用於公文指定受文者、審核人等下拉選單

**更新個人資料（`PATCH /users/me`）：**
- 可更新：`display_name`、`student_id`、`phone`、`show_phone`、`show_email`

**補完資訊頁（前端 `/profile/complete`）：**
- 首次登入時引導填入學號、電話等資訊

## 三、組織架構與職位管理

### 3.1 組織樹

組織採用**樹狀結構（Adjacency List）**，支援無限層級的父子組織。

**查看組織（`GET /orgs/tree`）：**
- 回傳完整樹狀組織結構
- 每個節點包含：`id`、`name`、`description`、`prefix`（字號前綴）、子節點列表

**組織 CRUD：**

| 操作 | 端點 | 所需權限 |
|------|------|---------|
| 建立組織 | `POST /orgs` | `org:manage` 或 `admin:all` |
| 查看所有組織（扁平清單） | `GET /orgs` | 已登入 |
| 查看組織樹 | `GET /orgs/tree` | 已登入 |
| 查看單一組織 | `GET /orgs/{org_id}` | 已登入 |
| 更新組織 | `PATCH /orgs/{org_id}` | `org:manage` 或 `admin:all` |
| 刪除組織 | `DELETE /orgs/{org_id}` | `org:manage` 或 `admin:all` |

**組織欄位：**
- `name`：組織名稱（如「學生代表大會」、「學務部」）
- `prefix`：公文字號前綴（如「嶺代」）
- `parent_id`：上級組織 ID（根節點為 null）
- `description`：描述

**查詢有公文起草權的組織（`GET /orgs/my-create-orgs`）：**
- 僅回傳當前使用者有 `document:create` 權限的組織

### 3.2 職位管理

每個組織可設有多個職位，職位負責承載**權限碼**。

**職位 CRUD：**

| 操作 | 端點 | 所需權限 |
|------|------|---------|
| 建立職位 | `POST /orgs/{org_id}/positions` | `org:manage_positions` 或 `admin:all` |
| 查看組織職位 | `GET /orgs/{org_id}/positions` | 已登入 |
| 查看職位詳情 | `GET /positions/{position_id}` | 已登入 |
| 更新職位 | `PATCH /positions/{position_id}` | `org:manage_positions` 或 `admin:all` |
| 刪除職位 | `DELETE /positions/{position_id}` | `org:manage_positions` 或 `admin:all` |

**職位欄位：**
- `name`：職位名稱（如「主席」、「秘書長」）
- `weight`：排序權重（數字越大越優先顯示）
- `permissions`：權限碼列表

### 3.3 使用者任期管理

**任期（UserPosition）** 記錄「哪位使用者在哪段時間內擔任哪個職位」。

**任期 CRUD：**

| 操作 | 端點 | 所需權限 |
|------|------|---------|
| 新增任期 | `POST /user-positions` | `admin:all` |
| 查詢全部任期 | `GET /user-positions` | `admin:all` |
| 查詢自己任期 | `GET /user-positions/me` | 已登入 |
| 更新任期（如卸任） | `PATCH /user-positions/{up_id}` | `admin:all` |
| 刪除任期 | `DELETE /user-positions/{up_id}` | `admin:all` |

**任期欄位：**
- `user_id`：使用者
- `position_id`：職位
- `start_date`：就任日期
- `end_date`：卸任日期（null 表示現任）

**時間有效性：** 系統查詢權限時自動套用日期過濾，確保只有任期有效期間內的使用者才擁有對應權限。

## 四、RBAC 權限系統

### 4.1 權限運作機制

```
組織（Org）
 └── 職位（Position）
      ├── 權限碼（Permission code，如 document:create）
      └── 使用者任期（UserPosition）
           ├── 使用者（User）
           └── 任期區間（start_date / end_date）
```

**查詢邏輯（每次 API 請求）：**
1. 解析 JWT 取得 `user_id`
2. 查詢該使用者今日有效的所有任期
3. 取得所有有效任期對應職位的所有權限碼
4. 以 `frozenset[str]` 形式傳遞至 API 依賴項

### 4.2 兩種權限範圍

| 類型 | 說明 | 範例 |
|------|------|------|
| **全域（Global）** | 對整個系統有效 | `admin:all`、`finance:view` |
| **組織級（Org-scoped）** | 僅對特定組織有效 | `document:create`（A 組織的秘書長只能建立 A 組織的公文） |

## 五、公文系統

### 5.1 公文種類

| 類別代碼 | 中文名稱 | 適用情境 |
|---------|---------|---------|
| `decree` | 令 | 機關首長命令 |
| `letter` | 函 | 機關間往來文書 |
| `announcement` | 公告 | 公開周知事項 |
| `report` | 報告 | 上呈長官的業務報告 |
| `meeting_notice` | 開會通知單 | 會議通知 |
| `other` | 其他 | 其他類型 |

### 5.2 速別（緊急程度）

| 代碼 | 名稱 | 說明 |
|------|------|------|
| `express` | 最速件 | 最緊急，優先處理 |
| `priority` | 速件 | 一般緊急 |
| `normal` | 普通件 | 標準速度 |

### 5.3 公文狀態機

```
草稿（draft）
    │
    ├─ submit（送審）──────────────→ 待審核（pending）
    │                                    │
    ├─ issue-direct（逕行發文）          ├─ approve（核准）
    │         │                          │    ├─ 尚有下一關 → 繼續 pending（通知下一位）
    │         ▼                          │    └─ 最後一關 ──→ 已核准（approved）
    └─────→ 已核准（approved）           │
                │                        ├─ reject（退件）
                ├─ archive（封存）        │    ├─ to_creator → 已退件（rejected）【終態】
                ▼                        │    └─ to_previous → 退回上一關（繼續 pending）
            已封存（archived）【終態】   │
                                         └─ recall（撤回）──→ 草稿（draft）

已退件（rejected）【終態，不可再操作】
```

### 5.4 公文 CRUD 操作

| 操作 | 端點 | 所需權限 | 說明 |
|------|------|---------|------|
| 建立草稿 | `POST /documents` | `document:create`（組織級） | 自動分配字號 |
| 列出公文 | `GET /documents` | 已登入 | 自動過濾至有權查看的公文 |
| 查看公文詳細 | `GET /documents/{doc_id}` | 有訪問權 | 含簽核步驟、附件、受文者 |
| 編輯草稿 | `PATCH /documents/{doc_id}` | `document:edit` + 建立者 | 僅草稿狀態可編輯 |

**列出公文的過濾參數：**
- `status`：草稿/待審/已核准/已退件/已封存
- `org_id`：指定組織
- `category`：公文類別
- `keyword`：標題關鍵字搜尋
- `pending_my_review`：僅顯示等待我審核的公文

### 5.5 送審與簽核流程

**送審（`POST /documents/{doc_id}/submit`）：**
- 所需權限：`document:submit`
- 建立多層 DocumentApproval（依設定的簽核流程）
- 狀態：草稿 → 待審核
- 自動通知第一位審核人（Email + WebSocket + 站內通知）

**核准（`POST /documents/{doc_id}/approve`）：**
- 所需權限：`document:approve`
- 確認當前請求者是否為輪到的審核人
- 若尚有下一關：推進至下一位審核人，通知其審核
- 若為最後一關：狀態改為已核准，通知承辦人

**退件（`POST /documents/{doc_id}/reject`）：**
- 所需權限：`document:reject`
- 兩種退件模式（`reject_mode` 參數）：
  - `to_creator`（預設）：退回至承辦人，狀態變為已退件，流程終止
  - `to_previous`：退回至上一位審核人，狀態仍為待審核，流程繼續

**撤回（`POST /documents/{doc_id}/recall`）：**
- 所需權限：`document:recall`
- 條件：第一關尚未開始審核
- 狀態：待審核 → 草稿

**逕行發文（`POST /documents/{doc_id}/issue-direct`）：**
- 所需權限：`document:issue_direct`（限機關首長）
- 跳過所有審核步驟，直接核准
- 狀態：草稿 → 已核准

**封存（`POST /documents/{doc_id}/archive`）：**
- 所需權限：`document:archive`
- 條件：公文必須已核准
- 狀態：已核准 → 已封存（終態）

### 5.6 附件管理

| 操作 | 端點 | 說明 |
|------|------|------|
| 列出附件 | `GET /documents/{doc_id}/attachments` | |

**更新受文者（`PUT /documents/{doc_id}/recipients`）：**
- 整批更新（replace all）

**受文者類型：**
| 類型 | 說明 |
|------|------|
| `main` | 主旨對象（函類的「受文者」）|
| `primary` | 正本收文單位（需執行） |
| `copy` | 副本收文單位（知悉備查） |

### 6.1 法規狀態流程

```
草稿（draft）
    │
    ├─ publish（簡易發布）──────────────────────────────────→ 已發布（published）
    │                                                                │
    └─ submit（送審）──→ 審議中（under_review）                     ├─ archive（停用）
                              │                                      ▼
                         schedule（排入議程）               已停用（archived）【終態】
                              │
                              ▼
                         排程中（scheduled）
                              │
                         council_approve（議會核定）
                              │
                              ▼
```

### 8.4 樂觀鎖防超賣機制

使用 SQLAlchemy `version_id_col` 實作樂觀鎖：

1. 讀取商品時取得當前版本號
2. 下單時附帶版本號更新庫存
3. 若版本號不符（已被他人搶先購買）→ 回傳 `409 Conflict`
4. 客戶端收到 409 可選擇重試

**特點：** 無需資料庫鎖定（lock-free），適合高並發搶票場景。

## 十六、優缺點分析

### 16.1 優點

#### 架構設計
- **Async-first 設計**：全面採用 asyncio，I/O 密集操作效能優異，可同時處理大量 WebSocket 連線
- **嚴格分層架構（Router → Service → Model）**：職責清晰，降低耦合，便於測試與維護
- **五權分立哲學**：以真實治理邏輯劃分模組邊界，系統設計貼合業務語義
- **Monorepo 架構（uv Workspaces）**：前後端與共用函式庫統一管理，依賴版本一致

#### 安全性
- **40 個細粒度權限碼**：比角色制（RBAC-role）更精確，可精準控制每個操作
- **組織級權限（Org-scoped）**：防止跨組織越權，天然的資料隔離
- **時間有效任期**：自動處理人員異動，歷史紀錄完整保留
- **JWT Token Rotation**：每次刷新後舊 Token 自動失效，防止 Token 被盜用後長期有效
- **IDOR 防護**：學餐、購票等跨資源操作均有所屬驗證
- **樂觀鎖（Optimistic Locking）**：購票系統防超賣，無鎖定開銷

#### 功能完整性
- **三渠道通知（站內 + Email + WebSocket）**：確保重要事件不被遺漏
- **多格式報表匯出（Excel + CSV）**：支援各種後續處理需求
- **公文列印功能**：HTML 格式輸出，直接使用瀏覽器列印或存 PDF
- **法規條文版本歷程**：每次修改有快照，可追溯歷史版本
- **自動結單（Celery Beat）**：學餐系統無需手動操作

#### 開發體驗
- **Swagger UI 文件**：自動生成，開發與測試便利（`/docs`）
- **Alembic 遷移管理**：資料庫 Schema 變更有版本控制
- **真實資料庫測試**：測試可信度高，不會出現 mock 與 production 行為差異

---

### 16.2 缺點與限制

#### 功能限制
- **僅支援 Google 登入**：無法使用帳號密碼登入，對無 Google 帳號的使用者造成障礙
- **附件儲存為本地檔案系統**：無雲端儲存（S3 等），擴展性受限，無備份機制
- **公文僅支援有限附件格式**（PDF/JPG/ZIP）：無法直接上傳 Word、Excel 等辦公文件
- **報表功能較簡單**：缺乏進階圖表分析、跨模組報表

#### 使用者體驗
- **學餐每人每排程限一筆**：無法一次訂多份（如代訂）
- **退件後無法重新送審**：公文退件為終態，必須重新建立新公文
- **法規搜尋僅全文搜尋**：缺乏結構化搜尋（如依條號、修訂日期範圍）
- **問卷題型無附圖功能**：無法在題目或選項中附上圖片

#### 技術限制
- **購票系統無付款整合**：缺少金流（線上付款）功能，需線下收費
- **LINE Bot 功能尚待擴充**：目前僅有 Webhook 接收基礎架構，主動推播功能需擴充
- **無行動應用**：目前僅有網頁版，無 iOS/Android App
- **WebSocket 為單機架構**：多台 API 伺服器時需要 Redis Pub/Sub 或其他廣播機制才能支援（目前未實作）
- **無前端搜尋快取**：大型公文/法規清單每次都需要後端查詢

#### 維運
- **無監控儀表板**：缺乏 Prometheus/Grafana 等監控整合
- **附件無病毒掃描**：上傳的 ZIP 等檔案未掃毒
- **無多語系（i18n）支援**：系統僅支援繁體中文介面
