## 一、系統總覽

### 1.1 系統定位

CKSC Legislation 是**建國高中班代大會**的法制資訊平台，主要提供：
- 校內自治組織的**法令公開閱覽**（憲章、法律、命令）
- **公文的起草、發布與查閱**
- **評議委員會的司法文書**管理
- 供任何人提交的**修法草案流程**
- 後台的**帳號與郵寄清單管理**

### 1.2 技術架構

| 層次 | 技術 | 說明 |
|------|------|------|
| 前端框架 | Vue 3 + Quasar Framework | Material Design UI |
| 後端/DB | Google Firebase Firestore | NoSQL 即時資料庫 |
| 雲端函式 | Firebase Cloud Functions | 伺服器端業務邏輯（發布通知等） |
| 檔案儲存 | Firebase Storage | 附件上傳 |
| 全文搜尋 | Algolia InstantSearch SSR | 法令關鍵字即時搜尋 |
| 身份驗證 | Firebase Auth（Google 登入） | |
| 渲染模式 | SSR（Server-Side Rendering）| 法令、公文頁面支援 SEO |
| 部署平台 | Google App Hosting + Firebase Hosting | |

## 二、法令系統（公開瀏覽）

### 2.1 法令清單頁（`/legislation`）

**任何人不需登入即可使用。**

#### 功能：

1. **Algolia 全文即時搜尋**
   - 在搜尋框輸入關鍵字，即時顯示符合的法令
   - 搜尋結果高亮顯示（黃色標記）匹配的條文關鍵字
   - 支援 SSR（伺服器端渲染），對搜尋引擎友好

2. **類別篩選側欄**
   - 點擊類別圖示篩選，再次點擊取消篩選
   - 支援的法令類別：

| 類別代碼 | 中文名稱 | 類型 |
|---------|---------|------|
| `CO` | 憲章 | 憲章（最高位階） |
| `CH` | 主席與副主席 | 法律 |
| `ED` | 行政部門 | 法律 |
| `SC` | 班代大會 | 法律 |
| `JC` | 評議委員會 | 法律 |
| `EO` | 行政命令 | 命令 |
| `SCO` | 班代大會命令 | 命令 |
| `JCO` | 評議委員會命令 | 命令 |
| `VCO` | 選舉委員會命令 | 命令 |

3. **每筆法令的操作按鈕**
   - **檢視全文**：前往法令詳細頁
   - **複製連結**：複製直連網址到剪貼簿
   - **起草修正**：前往修法草案編輯頁（需登入才能送交；不登入也可編輯，但只能匯出）

### 2.2 法令詳細頁（`/legislation/:id`）

#### 功能：

1. **顯示法令全文**
   - 條文依類型分層渲染（編→章→節→款→條→SpecialClause）
   - SpecialClause（訴訟典用條文）預設展開，可收折
   - 已刪除條文顯示刪除線
   - 已凍結條文顯示警告標示，並附連結至凍結依據文件

2. **右側目錄導覽欄**
   - 顯示所有條文標題
   - 點擊任一條文標題即滾動跳轉至該條文
   - 已刪除條文顯示刪除線

3. **立法沿革表**
   - 依時間順序列出每次修法紀錄
   - 每筆沿革包含：日期、修法摘要
   - 若有內容快照，顯示：
     - **比較箭頭按鈕（compare_arrows）**：與上一版本比較差異
     - **合併按鈕（merge_type）**：與目前現行版本比較差異
   - 若有關聯公文，顯示**外部連結按鈕**

4. **版本差異比對對話框（`LegislationHistoryDiffDialog`）**
   - 點選比較按鈕後彈出
   - 手機版：逐行比較（line-by-line）
   - 桌機版：並排比較（side-by-side）
   - 字元級別差異高亮（新增用紅色底線、刪除用灰色刪除線）
   - 比對模式：
     - `previous`（預設）：與上一個版本快照比較
     - `current`：與目前現行法條比較

5. **附帶決議（Addendum）顯示**
   - 補充決議文的日期與條列內容

6. **附件顯示（`AttachmentDisplay`）**
   - 顯示文件附件清單，可內嵌預覽（列印模式下停用內嵌）

7. **列印功能**
   - 點擊列印按鈕使用瀏覽器列印 / 另存 PDF
   - 展開/折疊所有條文（方便列印前整理）

8. **失效法令提示**
   - 若法令已凍結（`frozenBy` 欄位有值），頁面最上方顯示「本法令已遭凍結或失效」警告，附凍結依據連結

9. **SEO 支援**
   - 法令名稱作為頁面 `<title>`
   - 支援 Hash 導航（`#條文index`）與 Query 參數（`?c=條文index`）跳轉至特定條文
   - `og:updated-time` 自動設為最後修法日期

## 三、修法草案系統

### 3.1 草案編輯頁（`/legislation/:id/amendment`）

**任何人（含未登入）皆可使用草案編輯功能；送交審查需登入。**

草案資料儲存在**瀏覽器 LocalStorage**，關閉頁面後資料依然保留。

#### 三步驟流程（Stepper）：

---

**第一步：草案列表**

| 功能 | 說明 |
|------|------|
| 查看已儲存草案 | 顯示所有在本瀏覽器中已儲存的草案清單（名稱、類型、最後修改時間） |
| 新增草案 | 輸入草案名稱，選擇修正類型（部分/全文），從法規現行條文複製初始內容 |
| 重新命名草案 | 點擊鉛筆按鈕，輸入新名稱 |
| 繼續編輯 | 開啟已儲存草案進入編輯步驟 |
| 刪除草案 | 需確認，刪除後無法復原 |
| 匯入草案 | 上傳 `.ckla` 格式草案檔案（JSON），支援匯入後自動重導向至正確法規 |

**草案類型選擇**：
- **部分修正**：針對特定條文進行修改、新增、刪除，並附修正理由
- **全文修正**：重新撰寫整部法規的完整結構與內容

---

**第二步：編輯草案**

**（A）部分修正模式**

每個條文顯示目前狀態標籤（現行條文 / 新增 / 修正 / 刪除）：

| 操作 | 適用狀態 | 說明 |
|------|---------|------|
| **修改內容** | 現行條文、修正中 | 開啟條文編輯對話框，修改後顯示紅綠 diff |
| **標示刪除** | 現行條文 | 將條文標記為「刪除」（顯示刪除線） |
| **取消刪除** | 刪除中 | 恢復條文為現行狀態 |
| **復原修改** | 修正中 | 還原至原始條文內容（需確認） |
| **向下新增** | 任何位置 | 在特定位置後插入新條文 |
| **刪除新增** | 新增中 | 移除本次新增的條文（需確認） |
| **拖曳排序** | 全部 | 切換拖曳排序開關，拖動調整條文順序 |
| **填寫修正說明** | 非現行條文 | 每個被修改的條文可填入修正理由 |

**Inline Diff 顯示**：逐字元比較原始文字與修正後文字，紅底刪除 / 綠底新增。

**（B）全文修正模式**

| 操作 | 說明 |
|------|------|
| 新增內容區塊 | 新增條、章、節、編等各類型條文 |
| 編輯內容 | 開啟對話框修改任一區塊 |
| 刪除內容 | 移除一個內容區塊（需確認） |
| 向下新增 | 在指定位置後插入新區塊 |
| 拖曳排序 | 切換拖曳排序，調整區塊順序（自動重新計算 index） |

**條文類型選單**：
- 編（Volume）、章（Chapter）、節（Section）、款（Subsection）
- 條（Clause）、SpecialClause（訴訟典用，預設展開）

**自動生成標題**：系統自動偵測相同類型的現有條文，自動產生下一個序號（支援阿拉伯數字和中文數字）。

---

**第三步：提交與匯出**

| 操作 | 說明 |
|------|------|
| **進入草案送出程序** | 若已登入，前往送出確認頁（`/amendment/submit`） |
| **匯出草案檔（.ckla）** | 將草案資料輸出為 `.ckla` 格式 JSON 檔，可儲存或分享給他人 |
| **列印對照表（PDF）** | 產生「修正前 vs 修正後」對照表格，可列印或另存 PDF |

**.ckla 格式說明**：
- 自訂副檔名的 JSON 格式（`ckla = CKSC Legislative Amendment`）
- 包含：法規 ID、草案名稱、修正類型、修正內容、版本號
- 匯入時若法規 ID 不符，自動提示重導向至正確法規

---

### 3.2 草案送出頁（`/legislation/:id/amendment/submit`）

需**登入**才能送出。

| 顯示資訊 | 說明 |
|---------|------|
| 草案名稱 | 正在送出的草案名稱 |
| 聯絡人姓名 | 登入使用者的顯示名稱 |
| 聯絡信箱 | 登入使用者的 Email（確保具名可聯繫） |
| 預覽變更內容 | 可展開查看修正前後對照 |

送出後：透過 **Firebase Cloud Function** (`submitAmendmentRequest`) 提交草案，成功後清空草案記錄並返回法規頁面。

## 四、修法草案審查系統（後台）

### 路徑：`/manage/amendments/:id`

**需登入且具有對應法規類別的審查權限。**

採用**四步驟 Vertical Stepper**，所有進度**自動同步儲存至 Firestore**（防止意外中斷）：

---

**第一步：預覽變更內容**

- 顯示提案人姓名
- 顯示修正前後的完整對照表（`DraftAmendmentDiffPrint`）
- 查看無誤後，點擊「下一步」

---

**第二步：審核決定**

| 選項 | 說明 |
|------|------|
| **同意核可並發布** | 進入第三步，開始準備公文與立法沿革 |
| **退回草案** | 填寫退回理由（選填），確認後立即退回，系統通知提案人 |

---

**第三步：發布公文草稿**

- 系統自動生成本次修法的**公布令公文草稿內容**（依修正條文自動彙整）
  - 全文修正範例：「茲修正 XX 全文共 N 條，公布之。」
  - 部分修正範例：「茲增訂 XX 第一條、修訂第二條，公布之。」
- 審查者可修改：發文者姓名、公文內文
- 可選擇：
  - **直接在此頁建立公文草稿並繼續**
  - **儲存草稿至公文系統，使用進階編輯器**（另開新分頁，可新增附件等）
  - 若已儲存草稿，顯示公文編號連結

---

**第四步：立法沿革與發布**

- 系統依修正條文自動生成**立法沿革摘要**（如「公布修正第一條、第二條條文」）
- 審查者可手動修改摘要
- 確認後點擊「確定核准並發布」：
  - 呼叫 Firebase Function (`resolveAmendmentRequest`)
  - 系統自動更新法規的 `history` 陣列（新增一筆立法沿革）
  - 自動記錄本次修法的條文快照（儲存至 `historyContent` 子集合）
  - 自動發布公布令公文
  - 頁面灑彩帶（js-confetti）慶祝 🎉
  - 重導向至新建的公布令公文管理頁

**注意：** 若審查者對應的法規類別不符（如非主席無法核定主席令），系統回傳 403 並顯示「權限不足」。

## 五、公文系統（一般公文）

### 5.1 公文清單頁（`/document`）

**公開公文不需登入即可查看。**

#### 篩選條件（多維度複合篩選）：

| 篩選項目 | 說明 |
|---------|------|
| **公文字號** | 支援三種查詢模式：完整字號（如 `建班主公字第07810000001號`）、前綴（如 `建班主公字`）、號碼前綴（如 `079`） |
| **屆次** | 格式如 `79-1`（第 79 屆第 1 期），預設顯示當前屆次 |
| **發文日期範圍** | 「發文日期晚於」與「發文日期早於」，支援日曆選擇器 |
| **公文類型** | 見下方類型清單 |
| **發文部門** | 選擇大類（主席/行政部門/班代大會/評議委員會/特委） |
| **發文者** | 在選定部門中選擇具體身分（多選） |
| **受文部門** | 同上 |
| **受文者** | 同上（多選） |
| **已發布** | 已發布 / 未發布（僅後台管理模式可用） |

**管理模式（`manage` 模式）下的額外篩選：**

| 範圍選項 | 說明 |
|---------|------|
| 公開公文 | 所有已發布的公開公文 |
| 我起草的公文 | 僅限以自己 Email 為 `authorEmail` 的公文 |
| 分享給我的密件 | 有我的角色在 `viewers` 清單中的保密公文 |
| 所有我能檢視的公文 | 以上三類的聯集 |

**分頁載入：** 使用 Infinite Scroll，每次載入 10 筆，滾動到底自動載入更多。

**篩選條件同步至 URL Query String**：可書籤/分享特定查詢結果。

### 5.3 公文詳細頁（`/document/:id`）

公文字號格式：`{身分前綴}{類型字}字第{屆次}{期間次}{部門碼}{機關碼}{類型碼}{流水號}號`

- 範例：`建班主公字第07810000001號`
- 前三碼：屆次（079）
- 第四碼：期間次

## 十三、優缺點分析

### 優點

#### 架構與開發面

- **Firebase 全端整合**：Firestore + Auth + Storage + Functions 一站式，部署簡單，無需自建後端伺服器
- **Algolia 全文搜尋**：法令搜尋支援即時搜尋（毫秒級回應）、關鍵字高亮、Faceted 類別篩選，搜尋體驗優異
- **SSR 渲染**：法令和公文頁面支援伺服器端渲染，對搜尋引擎友好（SEO），且首次載入速度快
- **公文篩選強大**：支援 9 個維度的複合篩選（屆次、類型、發文者/受文者、日期範圍等），篩選條件同步 URL
- **本地草案儲存（LocalStorage）**：草案資料儲存在本地，關閉頁面也不會遺失；`.ckla` 匯出可分享協作
- **版本差異比對**：法令修法歷程支援逐字元 diff 比對，且可與「前一版本」或「目前版本」比較，功能完整

#### 功能面

- **完整的修法草案流程**：從起草（任何人）→提交→審查（4 步驟）→核可/退回，形成閉環
- **法令凍結/失效機制**：條文或整部法令可標記為凍結（連結至凍結依據），歷史資料仍保留可查
- **司法文書獨立入口**：評議委員會文書、訴訟查詢有獨立的分類頁面，符合實際組織運作
- **郵寄清單 + 發布通知**：公文發布時自動寄送 Email，批次管理郵寄對象
- **公文保密機制**：公文可設為保密（Confidential），只有指定角色才能查看
- **審查進度自動儲存**：草案審查頁面的進度（填寫的公文草稿、沿革摘要等）自動同步至 Firestore，不怕中途中斷

---

### 缺點與限制

#### 設計面

- **角色（身分）硬編碼**：所有 35+ 個角色直接定義在程式碼 `models.ts` 中，新增或修改角色必須改程式碼並重新部署，**無法從後台動態管理**
- **無細粒度權限控制**：後台功能（管理法令、管理公文、審查草案）的存取控制相對粗糙，主要依賴 Firebase Auth 判斷是否登入，未做細粒度的功能性權限分離
- **法令類別硬編碼**：9 個法令類別（憲章/主席/行政部門/班代大會/評委/各命令）固定寫死，無法動態新增新類別

#### 技術面

- **無訊息通知系統**：沒有站內通知、WebSocket 推播等；草案審查結果只依賴 Email，容易遺漏
- **Algolia 需要付費維護索引**：法令資料需手動或自動同步至 Algolia，有額外成本和同步問題
- **Firestore 1MB 文件限制**：法令內容存在單一文件，條文極多時有大小限制風險（系統透過 `historyContent` 子集合分離快照來緩解）
- **無離線支援**：公文和後台管理不支援離線操作（草案編輯透過 LocalStorage 有限支援）

#### 使用體驗面

- **修法審查流程只有 Email 通知**：草案送交後，審查者僅靠 Email 得知，沒有後台「待審清單」，容易漏接
- **帳號管理較簡單**：帳號 CRUD 功能存在，但沒有角色列表、活躍任期等進階管理，不適合頻繁換屆
- **沒有公文簽核流程**：公文系統只有「起草→發布」兩個狀態，沒有多層審核、退件、送簽等完整公文流程，功能較基礎
- **決議文搜尋非即時**：決議文需按下搜尋才過濾，而法令清單是即時搜尋，體驗不一致
- **無行動裝置最佳化版本**：雖 Quasar 有 RWD 支援，但部分操作（如多維篩選、條文拖曳）在手機上體驗較差

#### 維運面

- **無審計軌跡**：Firestore 直接寫入，沒有操作日誌（誰在什麼時間改了什麼法令），發生問題難以追溯
- **郵件寄送沒有佇列管理**：發布公文後透過 Firebase Function 同步寄送，若 Email 服務異常沒有重試機制
