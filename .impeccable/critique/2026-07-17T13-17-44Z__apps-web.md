## Anti-Patterns Verdict

**LLM assessment：中度產品型 AI slop，並非整體失控。** 角色化首頁、待辦排序、最近開啟、草稿復原與批次操作都反映真實產品思考；問題是任務型介面疊加了過量裝飾與入口。首頁 glow、`LIVE` pulse、Sparkles、卡片上浮、按鈕光掃與每頁淡入沒有幫助完成工作，反而增加視覺噪音。更嚴重的是同類控制項與設計 token 未完全統一。

**Deterministic scan：共 10 個 warnings。** `side-tab` 7 個、`bounce-easing` 2 個、`layout-transition` 1 個；其中 5 個可直接處理、4 個是 blockquote／勾號造型的 false positives、1 個選舉慶祝動畫屬情境判斷。

- 可處理：`admin/loans/records/page.tsx:317` 的單側 danger accent、`globals.css:2121` 的 `transition: width`、`globals.css:2143` 的永久 bounce、`globals.css:2172` 的 `.gold-line`、`globals.css:3257` 的 dashboard task 單側粗框。
- False positives：`globals.css:1761`、`:2087` 與 `RichTextarea.tsx:36` 是 blockquote；`globals.css:4374` 是用 border 畫勾號。
- 情境判斷：`globals.css:4481` 的選舉 winner-pop 使用 overshoot easing，雖是短暫慶祝，仍應尊重 reduced motion。

沒有可靠的瀏覽器 overlay。瀏覽器控制所需的可變 runtime 未暴露，因此沒有啟動 dev server 或 live server，也沒有宣稱頁面已完成視覺注入；視覺結論以原始碼與 deterministic scan 為依據。

## Overall Impression

這個平台不是缺少「智慧」，而是智慧功能被完整模組目錄與 mega-form 掩蓋。最大的機會是把導覽從「選模組」改成「完成一件事」：首頁只回答下一步、公文先問意圖、手機把主要操作固定在拇指區。現有草稿、最近項目、角色權限與 command menu 足以支撐這個改造，不需要重寫後端。

認知負荷檢查 8 項中失敗 6 項：single focus、chunking、visual hierarchy、one thing at a time、minimal choices、progressive disclosure。Grouping 與 working-memory support 相對良好。

明確超過 4 個選項的決策點包括：

- 預設側欄群組有 5、7、5 個入口，另有兩組管理入口；學生視角仍是 5＋5。
- Backoffice 最多 11 張同權重工具卡。
- Dashboard 同時展示最多 5 件待辦、4 件 matters、6 個 quick actions、5–6 個 services、最近開啟與 widget grid。
- 公文列表同時有 4 個快捷篩選、6 個狀態 tab、搜尋、進階篩選與儲存查詢；進階篩選約 14 個 state 欄位。
- 問卷提供 10 種題型，並同時展開必填、媒體、選項、格式、驗證與條件邏輯。
- 商品手機頁依序展示可持續增長的主題、系列與商品橫向清單。

## What's Working

1. **角色與權限導向已有正確骨架。** Dashboard primary action、quick actions、services 與行動版 tabs 會依角色調整，不是所有人看同一套功能。
2. **中斷復原能力適合手機。** 公文、附件與問卷有本地草稿復原，符合通勤中被打斷、切換 App 後返回的真實情境。
3. **真正的效率工具已存在。** 常用篩選、最近收件人、批次公文、最近開啟、Ctrl-K 與持久化導覽偏好，都是「更智慧」體驗的基礎。

## Priority Issues

### [P1] 首頁與全站 IA 讓智慧推薦淹沒在重複入口中

**Why it matters：** Dashboard 把推薦待辦、quick actions、service directory、matters、admin、recents 與 widgets 連續呈現；「待辦／工作／工作中心／整合工作台」語意重疊。手機使用者需要先比較入口，才能做原本只想快速完成的一件事。

**Fix：** 手機首頁只保留一個「下一件該做的事」、最多三個角色捷徑、最近項目與全站搜尋；其餘收進「全部服務」。側欄以使用情境重新分組，避免模組名稱同時充當工作流。

**Suggested command：** `$impeccable distill apps/web/src/app/dashboard`

### [P1] 公文與問卷是 mega-form，不是引導式任務

**Why it matters：** 公文一次展示內容、收件人、連結、附件、字號、可見度、承辦人與期限；問卷同時展示對象、題目、條件、驗證與發布。沒有完成度、review step 或手機 sticky action，錯誤會迫使使用者長距離回找。

**Fix：** 先以意圖或範本詢問 2–3 個問題，再逐步揭露欄位；提供步驟狀態、持續可見的「已儲存」、固定底部「儲存／下一步」，最後用摘要頁確認發布。把公文「設定審核人並送審」納入同一個可預期流程。

**Suggested command：** `$impeccable shape 公文與問卷的手機引導式建立流程`

### [P1] 高頻手機控制項未一致達到 44×44px，且部分互動不能鍵盤操作

**Why it matters：** 公文 tab、快捷篩選、checkbox、複製／移除按鈕、最近收件人 chip、工作完成按鈕與商品規格按鈕偏小；公文排序使用 clickable `<th>`。這同時傷害單手操作、低視力、鍵盤與動作輔助使用者。

**Fix：** 建立互動元件最低尺寸規範，將 icon action、chip、tab、checkbox 與 table sort 統一為可 focus、可辨識、至少 44px 的元件；手機主要動作落在底部拇指區。

**Suggested command：** `$impeccable adapt apps/web`

### [P2] 設計 token、控制項與暗色文字對比不一致

**Why it matters：** 會議、學餐與商品引用未定義的 `--muted`、`--surface`、`--card-bg`；primary action 也有多種外觀。Dark theme 的 `--text-muted` 約 2.52:1、`--text-disabled` 約 1.43:1，對小字尤其難讀。

**Fix：** 統一 semantic token 與 button/input/dialog 字彙，修正暗色文字對比至 WCAG AA；先處理 10–12px 的提示、badge 與表格次要資料。

**Suggested command：** `$impeccable audit apps/web`

### [P2] 錯誤復原與 contextual help 跟不上治理功能複雜度

**Why it matters：** 原生 prompt/confirm、泛用 toast、靜默 catch 與缺少決策點說明，讓新手在出錯後只能重試或詢問管理者。商品自製 dialog 缺 Escape 與 focus management，高風險發布也缺 review summary。

**Fix：** 將錯誤放回欄位或操作來源附近，保留輸入並提供下一步；用共享 ConfirmDialog／Drawer 取代原生 prompt 與不完整 modal；為字號、密等、正副本、可見度與條件邏輯加入簡短的就地說明與範例。

**Suggested command：** `$impeccable harden apps/web`

## Persona Red Flags

### Casey（分心的手機使用者）

- 公文與問卷需長距離捲動，主要 action 未固定在底部。
- 多個 checkbox、chip 與 icon action 小於 44px。
- Dashboard 重複捷徑讓「快速做一件事」變成先比較入口。
- 正面訊號是 bottom tab bar 與草稿復原，應保留並延伸。

### Jordan（第一次使用者）

- 五秒內能看到許多入口，卻不清楚第一步是哪一個。
- 「字號、密等、正本／副本、可見度、quorum、大屏」缺少 inline 解釋。
- 公文儲存草稿後才說明還需到詳情設定審核人，形成完成預期落差。
- 沒有任務導向的就地幫助或範例式引導。

### Sam（依賴鍵盤、低視力或動作輔助）

- `<th onClick>` 排序不能用鍵盤觸發。
- 多個移除 icon button 缺少清楚 aria-label。
- Shop 自製 dialog 沒有 Escape／focus trap。
- Dark theme muted text 未達 WCAG AA，小字進一步放大問題。

## Emotional Journey

- **進站：** 角色化問候、「先處理這些」與下一步建議讓人感到系統理解自己。
- **第一個低谷：** hero 後接著出現捷徑、服務、活動、管理、最近與 widgets，使用者再次承擔分類工作。
- **開始公文／問卷：** 高風險工作缺步驟與完成度；手機焦慮隨捲動增加。
- **公文流程：** 「儲存草稿」在承辦人與限辦日期之前出現，容易誤以為已完成全部設定；完成後又揭露設定審核人的下一段流程。
- **交易流程：** 學餐以巨大取餐碼、成功訊息與「我的訂單」收尾，是平台最好的 peak-end 範例。

## Minor Observations

- BottomTabBar 的五個入口、文字標籤、44px、safe-area 與鍵盤開啟時隱藏做得成熟。
- ModuleTabs 的高度、右側 fade 與橫向捲動提示合理。
- 商品 grid 在手機固定兩欄，遇到長名稱或放大字體可能過密。
- 問卷 datetime 強制 dark color scheme，淺色主題可能出現不一致 native control。
- 會議建立流程本身精簡且記住上次組織，但每張會議卡仍同時競爭「設定／控制台／大屏連結」。
- 自動檢測所見的永久 bounce 與 width transition 應移除或改為 transform；blockquote 與勾號命中不應誤修。

## Questions to Consider

1. 如果首頁只能保留一個推薦動作、三個捷徑與一個搜尋，其餘內容真的會有人想念嗎？
2. 使用者是來操作模組，還是來完成一件校園事務？能否讓「辦一場會議」自動串起會議、公文、出席與通知？
3. 公文能否先選「我要發通知／開會／回覆來文」，再由系統決定類別、字號、密等與欄位？
4. 手機最常做的三件事是否能在 30 秒、單手、少打字的情況下完成？
