## 📄 一、 公文格式與資料結構 (Schema)
為了符合標準公文格式，系統需支援以下欄位輸入與存儲：

* **基礎屬性**：
    * **速別**：最速件、速件、普通件。
    * **密等/解密條件**：普通、密、機密。
    * **附件說明**：支援文字描述與多檔案上傳（PDF, JPG, ZIP）。
* **角色與聯繫資訊**：
    * **承辦人資訊**：姓名、所屬單位（如：學聯會秘書處）、聯絡電話、電子郵件。
    * **發文機關**：組織全銜（如：國立〇〇大學學生自律委員會）。
* **收文者設定**：
    * **受文者**：主旨對象。
    * **正本**：需送達並執行的單位。
    * **副本**：僅供知悉或備查的單位。
* **內容架構**：
    * **主旨**：精簡描述公文目的。
    * **說明**：詳細事由、依據、背景。
    * **辦法**：具體要求的行動或執行方式（可選）。
* **時間戳記**：
    * **發文日期**：正式核定日期。
    * **限辦日期**：提醒審核者或受文者的截止時間。

## ⚖️ 二、 審核與簽核流程 (Workflow)
公文必須經過權限驗證與階層審核，方能生效：

* **草稿管理**：
    * **暫存功能**：支援隨時保存，不觸發審核。
    * **版本紀錄**：紀錄每次修改內容，防止爭議。
* **多級簽核鏈**：
    * **自定義流程**：根據公文類別自動帶入流程（例：一般公文需經組長→秘書長→會長）。
    * **核稿/判行**：審核者可進行「修改建議」或直接「核准」。
    * **退件機制**：審核者可退回至承辦人，並附上退件理由。
* **代理人設定**：
    * 當主管不在位時，可授權特定人員代為簽核，並註記「代」字。
* **數位印信掛載**：
    * 審核完成後，系統自動將組織電子圖記（大印）掛載至 PDF 指定位置。

## 🔔 三、 自動化通知機制 (Notification)
利用 Python 異步特性，確保相關人員即時掌握進度：

* **審核提醒**：
    * 當公文送至某位主管時，系統自動發送 Email 或 LINE 通知該主管「有待辦公文」。
* **狀態回報**：
    * 公文被「退件」或「核定成功」時，即時通知承辦人。
* **逾期催辦**：
    * 若公文在某一層級停留超過 48 小時且接近限辦日期，系統自動發送催辦通知。
* **收文知會**：
    * 正式發文後，系統自動發送公文電子檔（PDF）至受文者的聯絡郵件。

## 🚀 四、 發文與公開管理 (Publishing)
公文核定後的後續處理功能：

* **字號自動生成 (Auto-Numbering)**：
    * 依據組織代碼 + 年度 + 流水號自動產製（例：學聯(115)字第001號）。
    * **防重號鎖定**：確保多人同時發文時，字號不會衝突。
* **PDF 標準化輸出**：
    * 自動將資料填入標準公文模板。
    * 支援自動分頁、頁碼標註與浮水印防偽。
* **公文歸檔與檢索**：
    * **分類索引**：按年度、類別（令、函、公告）分類。
    * **全文檢索**：支援關鍵字搜尋主旨與說明內容。
* **公開查詢頁面**：
    * 前端提供公開列表，供全校學生查詢已發布的非機密公告，支援 SEO 優化。

## 🏛️ 學生自治組織權限組架構 (RBAC)

我們將權限分為 **「角色角色 (Role)」** 與 **「權限點 (Permission)」**。

### 1. 組織職位與字號權限表

| 組織名稱 | 角色 (Role) | 字號前綴範例 | 權限描述 |
| :--- | :--- | :--- | :--- |
| **班聯會** | `BAN_UNION_ADMIN` | 嶺聯 | 最高機關統稱 |
| **學生會** | `STUDENT_EXEC` | 嶺學 | 行政執行，負責多數一般行政公文。 |
| **學生議會** | `STUDENT_COUNCIL` | 嶺議 | 立法與預算審核，發布法規令。 |
| **評議委員會** | `JUDICIAL_COMM` | 嶺評 | 大法官職能，發布解釋文或裁決書。 |
| **學生代表團** | `REP_DELEGATION` | 嶺代 | 校務提案，發布對外建議書或提案。 |

以上都可以由該機關最高長官設置，且不一定只能一個，例如:嶺代議字、嶺代生字、嶺代霆字

### 2. 公文操作權限 (Permission Nodes)
* `doc.create`: 基本起草權限（各組成員）。
* `doc.issue`: **開立字號權限**（僅限各組最高長官：會長、議長等）。
* `doc.approve`: 簽核權限。
* `doc.view_all`: 查看組織內所有公文。

## 🔢 自動字號生成邏輯 (Serial System)

我們將字號拆解為三個部分：**[字號前綴] + [類別] + [年度] + [流水號]**。

### 字號結構範例：
> **嶺代生字第 1150000001 號**
* **嶺代**：由「學生代表團總召」預先開立的組織代碼。
* **生**：代表「生活輔導類」（長官預設多組供下屬選擇）。
* **115**：民國或西元年分（依學校慣例）。
* **0000001**：該字號下的唯一流水號。

### 技術實作：
我們在資料庫中使用 `Sequence`。你之前的錯誤就是因為我們正試圖建立這套自動化邏輯。
* 系統會檢查目前年度，若跨年則流水號自動重算（或持續累加）。
* 只有擁有 `doc.issue` 權限的長官可以建立新的「字號前綴組合」。

## 🔒 身分驗證與權限檢查流程

當一名使用者進入系統時：

1.  **Identity (身分驗證)**：
    * 使用 JWT Token 驗證，攜帶 `user_id`。
2.  **Assignment (所屬組別)**：
    * 系統查表確認該員屬於「學生會」或「學生代表團」。
3.  **Context Check (字號過濾)**：
    * 進入公文撰寫頁面時，系統僅會列出「所屬長官已開立」的字號供其選擇。
4.  **Action Check (操作檢查)**：
    * 若一般組員試圖「自創字號」，系統會噴出 `403 Forbidden`。

### **Module: Auth & Governance (身分與治理)**
* **目標**：確保各自治機關各司其職，不得越權發文。
* **功能需求**：
    1.  **組織樹管理**：支援四大機關獨立管理。
    2.  **動態字號池**：長官定義「字號模板」，組員填入「公文內容」。
    3.  **代行制度**：長官出缺時，支援由副手暫代開立字號之權限（即 Migration 中的 `is_acting` 欄位）。
