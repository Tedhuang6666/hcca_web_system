## Firestore Data Model

### Collection: `legislation`

Each document ID follows the pattern `{categoryIdPrefix}{zero-padded-number}`, e.g. `CO001`, `SC003`, `EO007`.

```
legislation/{id}
  name: string
  preface?: string
  category: string                  # LegislationCategory.firebase key
  createdAt: Timestamp
  frozenBy?: string                 # URL to the document that froze/voided this law
  content: LegislationContentRaw[]  # ordered array of content items
  history: LegislationHistoryRaw[]  # array of amendment history entries
  addendum?: AddendumRaw[]
  attachments?: Attachment[]
  resolutionUrls?: ResolutionUrl[]
```

### Subcollection: `legislation/{id}/historyContent/{contentId}`

Stores a content snapshot taken at the time a history entry was saved. Separated from the main document to avoid hitting Firestore's 1 MiB document size limit.

```
historyContent/{contentId}
  content: LegislationContentRaw[]  # snapshot of legislation.content at time of amendment
```

`contentId` format: `YYYYMMDD{counter}` in UTC+8 (e.g. `202603171`, `202603172`). Generated by `generateHistoryContentId()` in `src/ts/utils.ts`.

## State Management

### Legislation Cache (`src/stores/legislation.ts`)

`useLegislationStore` тАФ provides a client-side cache for legislation objects fetched on-demand (used by `SingleLegislationPage` to avoid re-fetching on SSR hydration).

| Member                | Description                                                                                   |
| --------------------- | --------------------------------------------------------------------------------------------- |
| `state.legislations`  | `Record<string, Legislation>` keyed by legislation document ID                                |
| `getLegislation(id)`  | Getter тАФ returns the cached `Legislation` or `null`; re-hydrates `Date` fields on each access |
| `loadLegislation(id)` | Action тАФ fetches from Firestore if not cached, stores result, returns `Legislation \| null`   |

### Draft Amendments (`src/pages/legislation/draft-amendment.ts`)

`useDraftAmendmentStore` тАФ Handles offline persistence for the draft amendment editor using VueUse's `useLocalStorage`.

| Member                 | Description                                                                       |
| ---------------------- | --------------------------------------------------------------------------------- |
| `state.amendmentType`  | `'partial' \| 'full' \| null` (Determines the editor mode used)                   |
| `state.partialContent` | `DraftContent[]` tracking individual modified clauses and user comments           |
| `state.fullContent`    | `LegislationContent[]` array containing the entirety of the rewritten legislation |
| `saveDraft(id)`        | Persists the entire state into `amendment_{id}` inside LocalStorage               |
| `clearDraft(id)`       | Erases the saved state for a specific legislation                                 |

## Pages

### `LegislationPage.vue` тАФ Public Law Index

- Uses **Algolia InstantSearch SSR** (`ais-instant-search-ssr`) for full-text search across law names and content.
- Left sidebar: filterable category list via `ais-menu` (`LegislationCategory.VALUES`).
- Results: each hit shows highlighted matching content with links to detail page.
- Accepts a `manage: boolean` prop тАФ when `true` (embedded in `ManageLegislationPage`), shows an "ч╖иш╝п" button pointing to the manage route.
- SSR state is passed through `useAlgoliaStore()` (session storage hand-off between server render and client hydration).

### `SingleLegislationPage.vue` тАФ Public Law Detail

- Fetches the legislation document via `useLegislationStore().loadLegislation(id)` on `onMounted`.
- Renders `LegislationContent` for each `content` item in index order.
- Right drawer: table of contents (clickable, scrolls to named anchor via `id = content.index`).
- Top history table: for each `LegislationHistory` with `contentId`, shows:
  - `compare_arrows` button тЖТ opens `LegislationHistoryDiffDialog` in `previous` mode (diff against the prior snapshot)
  - `merge_type` button тЖТ opens dialog in `current` mode (diff against live content)
  - `open_in_new` button тЖТ opens `history.link` (link to publishing document) if present
- Print support via `vue-to-print` (`useVueToPrint`).
- Hash/query navigation: scrolls to a specific clause on load if `#index` or `?c=index` is present.

**Key function:**

```ts
function openHistoryDiff(history: LegislationHistory, compareTarget: 'previous' | 'current' = 'previous');
```

Sets `selectedHistory`, `diffCompareTarget`, and opens the diff dialog.

### `DraftAmendmentPage.vue` тАФ Draft Preview / Editor

- Read-only base view of real legislation, augmented with a LocalStorage-backed draft editor.
- Implements a progressive stepper UI for selecting Amendment Type (Partial vs. Full), Editing, and Exporting.
- **Partial Editor**: Overlays inline edits on a per-clause basis. Allows tracking modifications, additions, and deletions with justification comments.
- **Full Editor**: Utilises `VueDraggable` to let users rewrite the entire hierarchical structure.
- **Exporting**: Users can export slimline JSON payloads of just their changes, or native Browser PDFs mapping the 'Before vs After' text layouts.

### `ManageLegislationPage.vue` тАФ Admin Law List

- Embeds `LegislationPage` with `manage` prop to show edit buttons in search results.
- Provides a "цЦ░хвЮц│Хф╗д" button that opens `LegislationDialog`.
- On submit: auto-generates the next document ID from the selected `LegislationCategory.idPrefix` + max existing counter + 1 (zero-padded to 3 digits), then calls `setDoc`.

### `ManageSingleLegislationPage.vue` тАФ Admin Law Editor

Full CRUD editor for a single legislation document. Uses `useLegislation(id)` for reactive two-way binding.

#### Content Management

| Function                 | Description                                                                                                                                                                                                        |
| ------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| `addContent(index?)`     | Opens the content form. If `index` is given, inserts **after** that index (`index + 1`). Otherwise appends at the end.                                                                                             |
| `editContent(content)`   | Populates form with existing content item.                                                                                                                                                                         |
| `submitContent()`        | For "add with insertBefore": bumps indices of all subsequent items, pushes new item, re-sorts, writes full `content` array. For "add at end": uses `arrayUnion`. For "edit": replaces by index, writes full array. |
| `removeContent(content)` | Prompts confirmation, then uses `arrayRemove`.                                                                                                                                                                     |
| `rearrange()`            | Called on drag-drop; writes the re-indexed full `content` array.                                                                                                                                                   |

Drag-drop reordering via `VueDraggable` (`vue-draggable-plus`). Toggle controlled by `draggable.content`.

**Content form fields:** type (select), title (auto-generated by `generateTitle()`), subtitle, body text, deleted checkbox, frozenBy URL, resolutionUrls list.

Auto-title generation (`generateTitle()`): scans existing content for the highest ordinal of the selected type and increments it. Uses `arabicOrdinal` to decide Arabic vs. Chinese numerals.

#### History Management

| Function                   | Description                                                                                                                                                                                                                                                             |
| -------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `addHistory()`             | Shows a **confirmation dialog** first (asking if content editing is complete), then opens the history form with `recordCurrent = true` defaulted.                                                                                                                       |
| `editHistory(history)`     | Populates form. Sets `recordCurrent = false`.                                                                                                                                                                                                                           |
| `submitHistory(skipCheck)` | If `recordCurrent` is true and content is non-empty: generates a `contentId` with `generateHistoryContentId()`, writes a snapshot to `historyContent/{contentId}`, stores `contentId` on the history object. Otherwise preserves the existing `contentId` (if editing). |
| `removeHistory(history)`   | Prompts confirmation, then uses `arrayRemove`. Note: the corresponding `historyContent` subcollection document is **not** automatically deleted.                                                                                                                        |

**`recordCurrentDisabled` computed:** The "Record current content" checkbox is disabled (with a warning hint) when editing a history entry that is not the most-recently-amended one. This prevents accidentally overwriting a historical snapshot with wrong data.

```
latestHistoryIndex = index of history with the highest amendedAt value
recordCurrentDisabled = historyAction === 'edit' && targetHistory.index !== latestHistoryIndex
```

#### Addendum Management

| Function                   | Description                                           |
| -------------------------- | ----------------------------------------------------- |
| `addAddendum()`            | Opens addendum form with today's date.                |
| `editAddendum(addendum)`   | Copies addendum into form.                            |
| `submitAddendum()`         | Add: `arrayUnion`. Edit: full `addendum` array write. |
| `removeAddendum(addendum)` | `arrayRemove`.                                        |

Addendum content is a `string[]` managed via `ListEditor` component.

#### Attachment Management

| Function                       | Description                                              |
| ------------------------------ | -------------------------------------------------------- |
| `addAttachment()`              | Opens attachment form.                                   |
| `submitAttachment()`           | Add: `arrayUnion`. Edit: full `attachments` array write. |
| `removeAttachment(attachment)` | `arrayRemove`.                                           |

Uses `AttachmentUploader` component for Firebase Storage uploads; resolves to URL strings pushed into `targetAttachment.urls`.

#### Legislation Metadata Edit

`edit()` / `submit()`: edits `name`, `category`, `createdAt`, `preface`, `frozenBy`, and `resolutionUrls` via `LegislationDialog`. Writes via `updateDoc`. `resolutionUrls` uses `deleteField()` when empty.

#### Shared Pattern

All mutation operations go through `submitProperty(determinant, addCallback, editCallback)`:

1. Shows `Loading` overlay
2. Dispatches to add or edit callback based on the `determinant` ref
3. Hides loading, shows success/error notification, nulls the `determinant`

## Components

### `LegislationContent.vue` тАФ Content Dispatcher

Reads `content.type` and renders the appropriate sub-component:

- `Clause` тЖТ `LegislationContentClause`
- `SpecialClause` тЖТ `LegislationContentClause` inside a `q-expansion-item` (collapsible); in print mode, renders inline
- `Volume` тЖТ `LegislationContentGeneric` (h4 bold)
- `Chapter` тЖТ `LegislationContentGeneric` (h5 bold)
- `Section` тЖТ `LegislationContentGeneric` (h6 bold)
- `Subsection` тЖТ `LegislationContentGeneric` (h6 bold)

Props: `content: LegislationContent`, `printing: boolean`, `expanded: boolean` (controlled `v-model`-style via `update:expanded` emit).

### `LegislationContentClause.vue`

Renders a numbered clause with title, subtitle (in `уАРуАС`), deleted state (strikethrough), frozen state, body text, line count, and resolution URLs.

### `LegislationContentGeneric.vue`

Simple heading renderer passed a CSS class for sizing. Used by Volume/Chapter/Section/Subsection.

### `LegislationAddendum.vue`

Displays an `Addendum` block (date + numbered string list). When `editable` prop is true, shows edit and delete buttons that emit `edit` and `remove` events to the parent.

### `LegislationDialog.vue`

Reusable dialog for creating or editing a legislation's top-level metadata:

- `name`, `preface`, `category` (select), `createdAt` (date picker)
- `frozenBy` URL toggle + input
- `resolutionUrls` list (add/edit/remove via nested `Dialog.create` prompts)

Uses `v-model` for the metadata object and emits `submit` / `canceled`. `validate()` method is exposed via `defineExpose` for parent to call before saving.

### `LegislationHistoryDiffDialog.vue` тАФ Diff Viewer

Shows a line-by-line or side-by-side diff between two legislation content snapshots.

**Props:**

| Prop              | Type                         | Description                                                   |
| ----------------- | ---------------------------- | ------------------------------------------------------------- |
| `modelValue`      | `boolean`                    | Dialog open state                                             |
| `selectedHistory` | `LegislationHistory \| null` | The history entry being inspected                             |
| `compareTarget`   | `'previous' \| 'current'`    | Whether to diff against prior snapshot or live content        |
| `sortedHistory`   | `LegislationHistory[]`       | Chronological history array for finding the previous snapshot |
| `currentContent`  | `LegislationContent[]`       | Live law content for `'current'` mode baseline                |
| `legislationId`   | `string`                     | Used to build the Firestore path for subcollection reads      |

**Lazy loading:**

Content snapshots are fetched on-demand when the dialog opens (via `watch` on `[modelValue, selectedHistory, compareTarget]`). Results are cached in `contentCache: Record<string, LegislationContent[]>` keyed by `contentId`. A spinner is shown while loading.

```
loadHistoryContent(contentId) тЖТ getDoc(legislation/{id}/historyContent/{contentId})
                               тЖТ maps each item through convertContentFromFirebase
```

**Baseline selection:**

- `'previous'` mode: walks backwards through `sortedHistory` from `selectedHistory`'s index to find the nearest entry with a `contentId`. If none found, baseline is empty (diff shows everything as added).
- `'current'` mode: baseline is `currentContent` (the live law).

**Text rendering (`formatSnapshot`):**

Each `LegislationContent` item is serialised to plain text lines:

```
чммNцвЭ уАРsubtitleуАС
чЛАцЕЛя╝Ъх╖▓хИкщЩд          (if deleted)
хЗНч╡Рф╛ЭцУЪя╝Ъ{url}       (if frozenBy)

{content body}

чЫ╕щЧЬц▒║шн░я╝Ъ
  Title: URL         (for each resolutionUrl)
```

**Responsive layout:**

- Mobile (`$q.screen.lt.md`): `line-by-line` (inline diff)
- Desktop: `side-by-side`

Uses `v-code-diff`'s `CodeDiff` component with `diff-style="char"` and `force-inline-comparison`.

### `InlineDiffRenderer.vue`

Computes and renders granular character-level text diffs natively. Used by `DraftAmendmentPage` and `DraftAmendmentDiffPrint` to highlight additions in red underline and deletions in grey strikethrough visually without large third-party widget overheads. Uses the `diff` library under the hood.

### `DraftAmendmentDiffPrint.vue`

A print-only styled component (`@media print`) for rendering the draft amendment comparison table. It outputs either a three-column line-by-line comparison (for partial edits) or a full document replacement view (for full edits). Implements forced page-margin padding and fake footers (`tfoot`) to ensure printed table rows break across A4 paper bounds without losing borders.

## Utility: `generateHistoryContentId` (`src/ts/utils.ts`)

```ts
generateHistoryContentId(refDate: Date, existingIds: string[]): string
```

1. Converts `refDate` to UTC+8 timezone.
2. Formats as `YYYYMMDD`.
3. Appends an auto-incrementing integer suffix (starting at `1`) until the result is not in `existingIds`.

Example: First ID on 2026-03-17 тЖТ `202603171`. A second on the same day тЖТ `202603172`.

## Firestore Security Rules Summary

From `firestore.rules`:

- `legislation/{legislation=**}` uses a wildcard that covers the `historyContent` subcollection.
- `allow read: if true` тАФ all legislation is publicly readable.
- `allow write: if isSignedIn()` тАФ requires Firebase Auth.

**Do not** change these rules without consulting the project's `firestore.rules` and verifying with the offline rules simulator. The wildcard intentionally covers the subcollection.

## Extension Points

- **New `ContentType`:** Add to `ContentType.VALUES` in `models.ts`, add a rendering branch in `LegislationContent.vue`, and handle it in `formatSnapshotContent` in `LegislationHistoryDiffDialog.vue`.
- **New `LegislationCategory`:** Add to `LegislationCategory.VALUES` in `models.ts`. The manage page auto-generates IDs from `idPrefix` тАФ ensure the prefix is unique.
- **History snapshot cleanup:** `removeHistory()` uses `arrayRemove` but does **not** delete the corresponding `historyContent/{contentId}` subcollection document. If storage cleanup is needed, add `deleteDoc(historyContentDocument(id, history.contentId))` inside `removeHistory()` when `history.contentId` is present.
